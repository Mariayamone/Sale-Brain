# [DEC-001]: Monolithic Express + Vite Architecture

**Date:** 2025-05-27  
**Status:** `Accepted`  
**Author:** Development Team  
**PR Reference:** N/A (Initial project setup)

## Context

The project required a full-stack JavaScript application with:
1. A frontend React dashboard for shop configuration
2. An API backend for state management and AI integration
3. Ability to run in development and production modes

We needed to decide on the server architecture that would:
- Serve the React frontend
- Handle API requests
- Integrate with Google Gemini AI
- Keep deployment simple (ideally single server)

## Decision

We chose a **monolithic architecture** where a single Express.js server (`server.ts`) handles:
- API endpoints (`/api/*`)
- Vite SSR middleware (for development)
- Static file serving (for production)
- Gemini AI integration

This means both frontend and backend run in the same Node.js process.

## Alternatives Considered

- **Option A: Separate Express + Vite dev server** — Running two separate servers (one for Express API, one for Vite dev server) would add complexity in development. Requires proxy configuration in Vite.

- **Option B: Next.js** — Using Next.js would provide built-in SSR/RSC capabilities and API routes. However, for this project, we preferred:
  - More control over the server architecture
  - Simpler setup since we don't need RSC
  - Lower learning curve for this specific use case
  - React 19 is already in use

- **Option C: Express + Nginx** — In production, separate static file serving via Nginx would be more performant, but adds infrastructure complexity. Not needed for MVP.

## Consequences

- **Positive:**
  - Single `npm run dev` starts both frontend and backend
  - Single `npm run build` and `npm run start` for production
  - Simpler deployment (one server to deploy)
  - Shared state between API and frontend in same process
  - Easy debugging (all code in one place)

- **Negative:**
  - Server restarts required for backend changes (using `tsx` for hot reload)
  - Can't scale frontend independently from backend
  - More memory usage than static file serving
