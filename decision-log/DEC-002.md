# [DEC-002]: File-based JSON State Persistence

**Date:** 2025-05-27  
**Status:** `Accepted`  
**Author:** Development Team  
**PR Reference:** N/A (Initial project setup)

## Context

The application needs to store:
- Shop configuration
- Product catalog
- Order history
- Customer sessions
- Delivery zones

We needed a persistence layer that was:
- Simple to implement
- Easy to inspect/debug
- Required no external database setup for development
- Could be later migrated to a proper database

## Decision

We chose **file-based JSON persistence** using `sales_brain_state.json` with:
- In-memory state object (`state: SystemState`)
- `loadState()` on server startup
- `saveState()` after state mutations
- JSON file on disk as source of truth

```typescript
// State Manager
let state: SystemState = { ...DEFAULT_STATE };

// Persistent Syncing
function loadState() {
  if (fs.existsSync(STATE_FILE)) {
    const parsed = JSON.parse(fs.readFileSync(STATE_FILE, "utf-8"));
    state = { ...DEFAULT_STATE, ...parsed };
  }
}

function saveState() {
  fs.writeFileSync(STATE_FILE, JSON.stringify(state, null, 2), "utf-8");
}
```

## Alternatives Considered

- **Option A: SQLite** — Would provide ACID compliance and better query capabilities. However:
  - Adds external dependency
  - More complex setup for dev
  - Overkill for initial MVP

- **Option B: Redis** — Great for session storage, but:
  - Requires Redis server running
  - Not ideal for complex object storage
  - Adds infrastructure requirement

- **Option C: Supabase (PostgreSQL)** — The long-term plan, but:
  - Requires Supabase account setup
  - More complex for initial development
  - API keys needed immediately

## Consequences

- **Positive:**
  - Zero external dependencies
  - Human-readable (can open JSON in editor)
  - Easy to backup (just copy the file)
  - No migration needed initially
  - Fast for small datasets

- **Negative:**
  - Not transactional (potential for partial writes)
  - No built-in concurrency handling
  - File can be corrupted (mitigated with try/catch)
  - Doesn't scale well (file size grows)
  - No real-time capabilities

## Rule Sync (if applicable)

This decision is reflected in the architecture doc and will be superseded when migrating to Supabase (future ADR).
